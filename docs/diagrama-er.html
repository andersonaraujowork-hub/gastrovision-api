Decisões de Design

Por que uma única tabela?

O enunciado exige dois tipos de usuário (OWNER e CUSTOMER) mas não impõe separação de tabelas. Uma única tabela com campo user_type atende todos os requisitos com menor complexidade, sem necessidade de joins e mais fácil de testar.

Por que @Embeddable no endereço?

O objeto Address é anotado com @Embeddable e incorporado em User via @Embedded. No banco gera colunas na mesma tabela, mas no código Java o endereço é um objeto separado — demonstrando separação de responsabilidades e boas práticas de OO, critério avaliado no trabalho.

Por que UUID como ID?

UUIDs evitam colisão em sistemas distribuídos e não expõem sequência numérica de registros. O JPA gera automaticamente via @GeneratedValue(strategy = GenerationType.UUID).

Por que created_at e updated_at automáticos?

As anotações @CreationTimestamp e @UpdateTimestamp do Hibernate preenchem os campos automaticamente, sem necessidade de lógica manual no service. O campo updated_at atende ao requisito de "data da última alteração" do enunciado.
